## Deep research contribution: BGP / Kubernetes LoadBalancer

New doc:
- `docs/bgp-kubernetes-load-balancing.md`

Validated sources:
- Kubernetes Service docs: <https://kubernetes.io/docs/concepts/services-networking/service/>
- MetalLB concepts: <https://metallb.io/concepts/>
- MetalLB configuration: <https://metallb.io/configuration/>
- MetalLB advanced BGP configuration: <https://metallb.io/configuration/_advanced_bgp_configuration/>
- Cilium BGP Control Plane: <https://docs.cilium.io/en/stable/network/bgp-control-plane/bgp-control-plane/>
- Cilium L2 Announcements / L2 Aware LB: <https://docs.cilium.io/en/stable/network/l2-announcements/>
- Calico advertise Kubernetes service IP addresses: <https://docs.tigera.io/calico/latest/networking/configuring/advertise-service-ips>
- Calico configure BGP peering: <https://docs.tigera.io/calico/latest/networking/configuring/bgp>

Observed:
- Kubernetes official Service docs define the Service abstraction and `LoadBalancer` type, but bare-metal environments need an additional controller/CNI/network integration to allocate and announce external IPs.
- MetalLB official docs describe IP pools plus L2 or BGP advertisements; advanced BGP configuration includes peer/node selection, service advertisement subset, source address, community aliases, and VRF-related options.
- Cilium official docs describe BGP Control Plane for advertising routes to connected routers; Cilium L2 Announcements are documented as Beta and include important constraints such as `externalTrafficPolicy: Local` incompatibility.
- Calico official docs describe BGP peering plus advertisement of service ClusterIP, ExternalIP, and LoadBalancer IP ranges; the docs call out ToR/ECMP/BGP multipath and service mode considerations.

Interpretation:
- For large on-prem Kubernetes, Service exposure should be reviewed as an IP ownership + route announcement + data-path problem, not only as a Kubernetes YAML problem.
- Design reviews should include IP pool governance, BGP/L2 ownership, ECMP behavior, source IP preservation, node drain behavior, failure detection, and observability.
- This is a generic public-doc synthesis only; it does not describe any specific company internal topology or router policy.

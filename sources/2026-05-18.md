## Deep research contribution: Kubernetes capacity planning / performance tuning

Validated sources:
- Kubernetes Resource Management for Pods and Containers: <https://kubernetes.io/docs/concepts/configuration/manage-resources-containers/>
- Kubernetes Reserve Compute Resources for System Daemons: <https://kubernetes.io/docs/tasks/administer-cluster/reserve-compute-resources/>
- Kubernetes Resource Quotas: <https://kubernetes.io/docs/concepts/policy/resource-quotas/>
- Kubernetes Limit Ranges: <https://kubernetes.io/docs/concepts/policy/limit-range/>
- Kubernetes Horizontal Pod Autoscaling: <https://kubernetes.io/docs/tasks/run-application/horizontal-pod-autoscale/>
- Kubernetes Node-pressure Eviction: <https://kubernetes.io/docs/concepts/scheduling-eviction/node-pressure-eviction/>
- Kubernetes Considerations for large clusters: <https://kubernetes.io/docs/setup/best-practices/cluster-large/>
- Kubernetes Metrics For Kubernetes System Components: <https://kubernetes.io/docs/concepts/cluster-administration/system-metrics/>
- Google SRE Book, Handling Overload: <https://sre.google/sre-book/handling-overload/>
- Google SRE Book, Addressing Cascading Failures: <https://sre.google/sre-book/addressing-cascading-failures/>

Observed:
- Kubernetes official resource-management docs validate the central role of CPU/memory requests and limits in scheduling and runtime enforcement.
- Kubernetes node allocatable docs validate reserving compute for kube/system daemons and eviction thresholds before treating remaining capacity as pod capacity.
- ResourceQuota and LimitRange docs validate namespace-level guardrails for aggregate resource use, defaults, min/max constraints, and object-count boundaries.
- HPA docs validate the autoscaling control loop, while also implying that autoscaling depends on accurate metrics and request sizing.
- Node-pressure eviction docs validate memory, filesystem/inode, and PID pressure as capacity red lines rather than after-the-fact cleanup trivia.
- Large-cluster and system-component metrics docs validate that control-plane capacity must be observed separately from workload compute capacity.
- Google SRE public chapters frame overload and cascading failures as reliability risks that capacity planning must explicitly reserve against.

Interpretation for k8s-info:
- Capacity planning should be presented as a layered model: SLO / workload profile / cluster resource envelope / control-plane envelope.
- Performance tuning should follow a low-risk loop: observe, hypothesize, change one layer, verify, and keep rollback.
- The guide avoids production tuning numbers and does not infer any private cluster size, SLO, node shape, cost, architecture, or incident detail.

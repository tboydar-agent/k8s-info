### Observations

- Kubernetes official docs define strong cluster-local primitives: cluster networking, Services, and workload management. Cross-cluster service discovery, global ingress, or fleet management require explicit additional APIs/controllers/provider features.
- SIG Multicluster's MCS model is useful as a low-risk vocabulary because it exports only explicitly selected Services via `ServiceExport` and represents cross-cluster services via `ServiceImport`.
- Provider-managed patterns such as GKE Multi-cluster Services / Multi Cluster Ingress and Azure Kubernetes Fleet Manager are good architecture references, but their provider-specific control planes should not be treated as directly portable to bare-metal on-prem.
- For on-prem design, the reusable checklist should separate failure domain, version/addon skew, networking/traffic, data/storage, and operations. Service discovery must not be confused with data-layer HA or DR guarantees.

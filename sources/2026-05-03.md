```text
  - kubelet: must not be newer than apiserver; may be up to 3 minors older
```

```text
For a v1.36 control plane, generic supported kubelet / kube-proxy planning range is 1.36 through 1.33,
subject to stricter provider, distro, addon, and HA apiserver constraints.
```

## 6) Deep-research contribution: Linux node pressure debugging

```text
Signal:
  Kubernetes node pressure should be triaged from kubelet eviction signals and node conditions,
  not only from kubectl top or generic Linux free/df output.

Core public facts observed:
  - node-pressure eviction is kubelet-local proactive Pod termination to reclaim node resources
  - eviction signals include memory.available, nodefs/imagefs/containerfs capacity and inodes, and pid.available
  - Node status exposes MemoryPressure, DiskPressure, PIDPressure, Ready, capacity/allocatable, and nodeInfo
  - the resource metrics pipeline provides basic CPU/memory metrics for autoscaling and kubectl top, but not complete filesystem/inode/PID/root-cause context
```

Sources:

- https://kubernetes.io/docs/concepts/scheduling-eviction/node-pressure-eviction/
- https://kubernetes.io/docs/reference/node/node-status/
- https://kubernetes.io/docs/tasks/debug/debug-cluster/
- https://kubernetes.io/docs/tasks/debug/debug-cluster/resource-metrics-pipeline/
- https://kubernetes.io/docs/concepts/configuration/manage-resources-containers/
- https://kubernetes.io/docs/concepts/scheduling-eviction/taint-and-toleration/

Planning impact:

```text
For large Linux node pools, a reusable pressure runbook should separate:
  - Kubernetes API evidence: node conditions, taints, events, pod status
  - kubelet/runtime evidence: eviction manager logs, CRI stats, image/log cleanup behavior
  - OS evidence: cgroup memory, filesystem capacity, inode pressure, process count
  - safety boundary: do not record internal hostnames, IPs, tickets, private links, or non-public incidents in public docs
```
